PHASE 01 What Are Databases, Really? No prior knowledge needed. Build the mental model first.
1.1
Databases vs Spreadsheets — The Mental Model
tables rows & columns relations why not Excel?

Why Not a Spreadsheet?

A spreadsheet is a flat file — all data lives in one grid. When data grows complex (orders tied to customers, products to categories), you either duplicate rows or lose integrity. There's no enforcement that a customer ID actually exists.

A relational database stores data in separate tables linked by keys. Each piece of data lives in exactly one place. Relationships are enforced by the engine itself — not by your discipline.

// Diagram — Database vs Spreadsheet

Core Mental Model

  • Table = a named grid of typed columns. Like a class/struct definition.
  • Row (tuple) = one record — one instance of that entity.
  • Column (attribute) = a field with a fixed data type. Every row must conform.
  • Relation = a link between tables via shared keys (PK → FK).
💡

The golden rule: store each fact once. If you find yourself copying the same customer name into 50 order rows, you need a customers table and a customer_id foreign key on orders.

1.2
Relational vs Non-Relational — When to Use What
RDBMS NoSQL ACID use cases

RDBMS vs NoSQL — The Real Decision

RDBMS (PostgreSQL)

Rigid schema, strict types, ACID transactions. Data has known structure and clear relationships. Ideal for: finance, SaaS apps, inventory, HR systems.

NoSQL (Mongo, Redis…)

Flexible schema, horizontal scale, eventual consistency. Data is document-like or graph-like with no fixed shape. Ideal for: caching, user-generated content, real-time feeds.

// Diagram — RDBMS vs NoSQL Decision Tree

ACID — What It Actually Means

  • Atomicity — A transaction is all-or-nothing. Transfer ₹500: both debit AND credit happen, or neither does.
  • Consistency — Every transaction takes the DB from one valid state to another. Constraints are never violated mid-transaction.
  • Isolation — Concurrent transactions don't see each other's uncommitted work. Prevents dirty reads.
  • Durability — Once committed, data survives crashes. PostgreSQL uses a Write-Ahead Log (WAL) to guarantee this.
🎯

In MNC interviews: "Use PostgreSQL when your data has relationships and you need ACID. Use MongoDB when your schema is dynamic and you need document-level flexibility." That's the clean answer.

How pgAdmin Relates to psql

psql is the CLI — fast, scriptable, ssh-friendly. pgAdmin is the GUI — good for visually exploring schemas. In production you'll use psql (or a migration tool). Master psql first; pgAdmin is a crutch.

1.4
Data Types — Choosing the Right Type for Every Column
INTEGER / BIGINT TEXT / VARCHAR BOOLEAN DATE / TIMESTAMP NUMERIC / DECIMAL UUID

Choosing the Right Data Type — Why It Matters

Wrong types cost you: storage, performance, correctness. TEXT where you need TIMESTAMPTZ means your app does date math in JavaScript instead of the DB. FLOAT for money means your balance sheet is wrong.

// Diagram — PostgreSQL Data Type Decision Map

Type Selection Rules (Interview Ready)

  • IDs: BIGINT GENERATED ALWAYS AS IDENTITY (preferred) or UUID for distributed systems.
  • Money: NUMERIC(12, 2) — exact, no float rounding.
  • Strings: TEXT over VARCHAR(n) unless you genuinely need a length cap enforced at DB level.
  • Timestamps: Always TIMESTAMPTZ — it stores UTC, displays in your session timezone.
  • Flags: BOOLEAN, not INTEGER (0/1 is a code smell).
1.5
CREATE TABLE — Schema Design from Scratch
PRIMARY KEY NOT NULL DEFAULT UNIQUE SERIAL / GENERATED

CREATE TABLE — Every Constraint Explained

CREATE TABLE users ( id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, -- GENERATED ALWAYS: PG manages this. You can't accidentally override it. -- PRIMARY KEY: implies NOT NULL + UNIQUE + creates a B-Tree index automatically. email TEXT NOT NULL UNIQUE, -- NOT NULL: row is rejected if email is missing. Never store NULL emails. -- UNIQUE: a separate B-Tree index is auto-created. Fast lookups by email. username VARCHAR(50) NOT NULL, is_active BOOLEAN NOT NULL DEFAULT TRUE, -- DEFAULT: value used when column is omitted in INSERT. created_at TIMESTAMPTZ NOT NULL DEFAULT now() );

When to Use SERIAL vs GENERATED ALWAYS AS IDENTITY

  • SERIAL is old shorthand — it creates a sequence and sets a DEFAULT. You CAN override it with an explicit value in INSERT. Risky.
  • GENERATED ALWAYS AS IDENTITY (PG 10+) is the SQL standard. The DB always controls the value. Safer. Use this.
  • GENERATED BY DEFAULT AS IDENTITY allows overriding — useful when you need to restore from a backup with known IDs.
🎯

After CREATE TABLE, run \d users in psql to verify column types, constraints, and the automatically-created indexes. This habit catches mistakes immediately.

CRUD — The Four Operations

INSERT INTO users (email, username) VALUES ('zaheer@dev.io', 'zaheer') RETURNING id, created_at; SELECT id, email FROM users WHERE is_active = TRUE; UPDATE users SET username = 'zaheer_dev' WHERE id = 1 RETURNING *; DELETE FROM users WHERE id = 1; -- ⚠ Always use WHERE on UPDATE/DELETE. Wrap in BEGIN; first in prod.

RETURNING is PostgreSQL-specific — it gives back affected rows after INSERT/UPDATE/DELETE. No need for a follow-up SELECT. Use it always.

1.6
Your First CRUD — INSERT, SELECT, UPDATE, DELETE
INSERT INTO SELECT * UPDATE … SET DELETE FROM WHERE clause

ON DELETE Options — Choose Carefully

  • CASCADE: Delete parent → delete children. Use for: orders/order_items, posts/comments.
  • SET NULL: Delete parent → FK becomes NULL. Use for: optional relations.
  • RESTRICT / NO ACTION: Prevents parent deletion if children exist. Use for: critical reference data.
CREATE TABLE orders ( id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE, total NUMERIC(12,2) NOT NULL CHECK (total >= 0) );

⚠️ Interview trap: "What's the difference between RESTRICT and NO ACTION?" RESTRICT checks immediately when the statement runs; NO ACTION defers check to end of transaction. In practice they behave the same for most use cases.

  DATE_TRUNC('month', created_at) AS month,
  status,
  COUNT(*)                         AS total_orders,

-- Find all customers + their orders (including customers with zero orders)
SELECT
  c.id,
  c.name,
  COUNT(o.id)      AS order_count,
  COALESCE(SUM(o.total), 0) AS lifetime_value

SELECT
  m.name                                      AS member,
  b.title,
  l.due_at,
  CURRENT_DATE - l.due_at                   AS days_overdue,
  0.50 * (CURRENT_DATE - l.due_at)           AS fine_usd

-- Lesson 3.3–3.4 · Window functions: ranking + running total + MoM growth

WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(amount)                      AS revenue
  FROM   orders
  GROUP  BY 1
)
SELECT
  month,
  revenue,
  SUM(revenue)  OVER (ORDER BY month)                       AS running_total,
  LAG(revenue)  OVER (ORDER BY month)                       AS prev_month,
  ROUND(
    100.0 * (revenue - LAG(revenue) OVER (ORDER BY month))
          / NULLIF(LAG(revenue) OVER (ORDER BY month), 0),
    2
  )                                                           AS mom_growth_pct
FROM  monthly_revenue
ORDER BY month;

-- Project 2 · Customer cohort retention analysis

WITH cohorts AS (
  SELECT
    id AS customer_id,
    DATE_TRUNC('month', created_at) AS cohort_month
  FROM customers
),
activity AS (
  SELECT
    o.customer_id,
    DATE_TRUNC('month', o.created_at) AS order_month
  FROM orders o
)
SELECT
  c.cohort_month,
  EXTRACT(MONTH FROM AGE(a.order_month, c.cohort_month)) AS month_number,
  COUNT(DISTINCT a.customer_id)                              AS active_customers
FROM      cohorts  c
LEFT JOIN activity a ON a.customer_id = c.customer_id
GROUP BY  1, 2
ORDER BY  1, 2;

-- Lesson 6.3 · Audit log trigger — track every change automatically

CREATE TABLE audit_log (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  table_name  TEXT        NOT NULL,
  operation   TEXT        NOT NULL,   -- INSERT | UPDATE | DELETE
  old_data    JSONB,
  new_data    JSONB,
  changed_by  TEXT        DEFAULT current_user,
  changed_at  TIMESTAMPTZ DEFAULT now()
);

CREATE OR REPLACE FUNCTION fn_audit_trigger()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  INSERT INTO audit_log (table_name, operation, old_data, new_data)
  VALUES (
    TG_TABLE_NAME,
    TG_OP,
    CASE WHEN TG_OP != 'INSERT' THEN row_to_json(OLD) END,
    CASE WHEN TG_OP != 'DELETE' THEN row_to_json(NEW) END
  );
  RETURN COALESCE(NEW, OLD);
END; $$;

CREATE TRIGGER tg_users_audit
  AFTER INSERT OR UPDATE OR DELETE ON users
  FOR EACH ROW EXECUTE FUNCTION fn_audit_trigger();

-- Lesson 6.4 · JSONB power — storing product metadata flexibly

CREATE TABLE products (
  id       BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name     TEXT  NOT NULL,
  metadata JSONB NOT NULL DEFAULT '{}'
);

-- GIN index for fast JSONB queries
CREATE INDEX idx_products_meta ON products USING GIN (metadata);

-- Insert product with flexible attributes
INSERT INTO products (name, metadata) VALUES
  ('MacBook Pro 16"', '{"brand":"Apple","ram_gb":32,"storage_tb":1,"tags":["laptop","pro"]}');

-- Query: find all products with >=16GB RAM that have "pro" tag
SELECT name, metadata ->> 'brand' AS brand
FROM   products
WHERE  (metadata ->> 'ram_gb')::INT >= 16
  AND  metadata -> 'tags' @> '["pro"]';
-- Lesson 6.5 · Full-text search with ranking

-- Add a generated tsvector column for performance
ALTER TABLE products
  ADD COLUMN search_vector TSVECTOR
    GENERATED ALWAYS AS (
      to_tsvector('english', COALESCE(name, '') || ' ' ||
                           COALESCE(metadata->>'description', ''))
    ) STORED;

CREATE INDEX idx_products_fts ON products USING GIN(search_vector);

-- Search with relevance ranking
SELECT
  name,
  ts_rank(search_vector, query)   AS rank,
  ts_headline('english', name, query) AS headline
FROM
  products,
  to_tsquery('english', 'laptop & pro') AS query
WHERE  search_vector @@ query
ORDER BY rank DESC;
🚀
PostgreSQL vs Elasticsearch for FTS

For most applications, PostgreSQL full-text search is sufficient and eliminates an entire infrastructure component. Only migrate to Elasticsearch when you need fuzzy multi-language search, advanced relevance tuning, or searching across hundreds of millions of rows.
