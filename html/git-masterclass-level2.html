Git Internals — Object Model DEEP DIVE
Understand what Git actually stores. Everything is an object.
🧬 Git's Content-Addressable Storage

Git stores 4 object types in .git/objects/ — each identified by a SHA-1 hash of its content. This is why Git is so reliable: if the content matches the hash, the data is intact. Blobs hold file content. Trees hold directory structure. Commits hold metadata + a pointer to a tree. Tags are annotated pointers to commits.

refs/heads/main ──────────────────────────────► SHA-1
└──► commit a1b2c3d {tree: x, parent: y, author, message}
└──► tree x9y8z7w {README.md → blob f1, src/ → tree g2}
├──► blob f1e2d3c "# My App\n..." ← file content
└──► tree g2h3i4j {index.js → blob k5}
└──► blob k5l6m7n "console.log('hi')" ← file content
bash — Exploring Git Object Internals
# Create a file and commit it, then inspect the objects
$ echo "hello git" | git hash-object --stdin -w

# See object type and size
$ git cat-file -t 8ab686

$ git cat-file -p 8ab686

# See the commit object for HEAD
$ git cat-file -p HEAD

# Inspect the tree (directory snapshot)
$ git cat-file -p HEAD^{tree}

# Count all objects in the repo
$ git count-objects -vH

size: 156.00 KiB
# Verify object integrity
$ git fsck

Checking connectivity: done.
git cat-file -p <hash>
Pretty-print any object (blob/tree/commit/tag). Your window into Git's storage.
git cat-file -t <hash>
Get the type of an object: blob, tree, commit, or tag.
git hash-object
Compute or write a hash for file content. The plumbing behind git add.
git ls-tree HEAD
List the tree for a commit — like ls but for git objects.
git rev-parse HEAD
Turn a ref (HEAD, main, v1.0) into a full SHA-1 hash.
git verify-pack -v

💡 Why it matters: Understanding objects explains why git rebase creates new commits (new hashes), why branches are just files in .git/refs/, and why Git never truly loses data within 90 days.

Git Hooks AUTOMATION
Trigger scripts automatically on Git events — lint, test, format, enforce.
🪝 What Are Hooks?

Hooks are shell scripts in .git/hooks/ that Git runs automatically at specific lifecycle events. They let you enforce code quality, run linters, validate commit messages, prevent accidental pushes to main, and automate deployment — all without a CI server.

CLIENT-SIDE HOOKS
pre-commit prepare-commit-msg commit-msg post-commit pre-push pre-rebase post-checkout post-merge
SERVER-SIDE HOOKS
pre-receive update post-receive
pre-commit
commit-msg
pre-push
Husky Setup
.git/hooks/pre-commit — Run linter before every commit
#!/bin/bash
# .git/hooks/pre-commit — runs before the commit is created
# If this script exits non-zero, the commit is ABORTED
# Run ESLint on staged JS files
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$')
if [ -n "$STAGED_FILES" ]; then
echo "🔍 Running ESLint on staged files..."
npx eslint $STAGED_FILES --max-warnings=0
if [ $? -ne 0 ]; then
echo "❌ ESLint failed. Fix errors before committing."
exit 1
fi
fi
# Also run Prettier check
npx prettier --check . --quiet || exit 1
echo "✅ Pre-commit checks passed!"
# Make the hook executable:
$ chmod +x .git/hooks/pre-commit
.git/hooks/commit-msg — Enforce Conventional Commits format
#!/bin/bash
# commit-msg hook — receives commit message file as $1
COMMIT_MSG=$(cat "$1")
PATTERN="^(feat|fix|docs|style|refactor|test|chore|perf|ci|revert)(\(.+\))?: .{1,72}"
if ! echo "$COMMIT_MSG" | grep -qE "$PATTERN"; then
echo ""
echo "❌ Invalid commit message format!"
echo " Required: feat|fix|docs|chore|refactor: description"
echo " Example: feat(auth): add JWT refresh token"
echo ""
exit 1
fi
echo "✅ Commit message format OK"
# Test it
$ git commit -m "bad message"
❌ Invalid commit message format!
$ git commit -m "feat: add login page"
✅ Commit message format OK
.git/hooks/pre-push — Run tests + block push to main
#!/bin/bash
# pre-push — protect main, run tests before any push
REMOTE="$1"
CURRENT=$(git symbolic-ref --short HEAD)
# Block direct push to main/master
if [[ "$CURRENT" == "main" || "$CURRENT" == "master" ]]; then
echo "🚫 Direct push to $CURRENT is not allowed!"
echo " Use a feature branch and open a PR."
exit 1
fi
# Run test suite before pushing
echo "🧪 Running tests..."
npm test -- --watchAll=false
if [ $? -ne 0 ]; then
echo "❌ Tests failed. Fix before pushing."
exit 1
fi
echo "✅ All good. Pushing..."
Husky — Share hooks with your team via npm
# Problem: .git/hooks are NOT committed. Team can't share them.
# Solution: Husky — manages hooks as committed files
$ npm install --save-dev husky
$ npx husky init
created .husky/pre-commit
# Edit .husky/pre-commit (this IS committed to git)
$ cat .husky/pre-commit
npx lint-staged
# Add lint-staged config to package.json
"lint-staged": {
"*.{js,ts,jsx,tsx}": ["eslint --fix", "prettier --write"],
"*.{json,md,css}": "prettier --write"
}
# Commit message validation with commitlint
$ npm install --save-dev @commitlint/{cli,config-conventional}
$ echo "export default {extends:['@commitlint/config-conventional']}" > commitlint.config.js
$ echo "npx --no commitlint --edit \$1" > .husky/commit-msg
# Now every dev on the team gets the same hooks on npm install!
⚡ Pro Setup: Husky + lint-staged + commitlint is the industry standard trio. Husky manages hook files, lint-staged only lints staged files (fast!), and commitlint enforces conventional commits.

Git Worktrees GAME CHANGER
Multiple working directories from ONE repository — no stashing, no switching.
🌳 The Worktree Problem

Ever needed to review a PR while mid-feature? Or test a hotfix without abandoning your current work? git worktree lets you check out multiple branches simultaneously in different folders — sharing the same .git database. No more stashing, no more cloning twice.

bash — git worktree
# You're on feature/dashboard, urgent hotfix needed
# Add a NEW working tree for hotfix WITHOUT switching branches
$ git worktree add ../hotfix-tree hotfix/payment-null
Preparing worktree (new branch 'hotfix/payment-null')
HEAD is now at a1b2c3d feat: dashboard grid
# Now you have two working directories:
~/projects/my-app/ ← feature/dashboard (original)
~/projects/hotfix-tree/ ← hotfix/payment-null (new)
# Work in the hotfix tree in a second terminal
$ cd ../hotfix-tree && vim src/payment.js
$ git commit -am "hotfix: null check in payment processor"
$ git push origin hotfix/payment-null
# Meanwhile your feature/dashboard code is UNTOUCHED
# List all worktrees
$ git worktree list
/home/user/projects/my-app a1b2c3d [feature/dashboard]
/home/user/projects/hotfix-tree d4e5f6g [hotfix/payment-null]
# Remove the worktree when done
$ git worktree remove ../hotfix-tree
# Or prune stale references
$ git worktree prune
💡 Pro Use Case: Use worktrees to run npm run dev on main (to see production) and npm run dev --port 3001 in your feature worktree simultaneously. Visual comparison without any switching!

Advanced Interactive Rebase HARD
Rewrite history like a surgeon — squash, fixup, reorder, edit, exec.
bash — git rebase -i — All Actions
$ git rebase -i HEAD~5
# The editor opens with your last 5 commits:
pick a1b2c3d feat: add login form
pick d4e5f6g fix: typo in label
pick g7h8i9j fix: another typo
pick j1k2l3m feat: add remember me checkbox
pick m4n5o6p chore: console.log cleanup
## ── Available actions ──
pick → keep commit as-is
reword → keep, but edit the message
edit → pause, amend, then continue
squash → combine with previous, edit message
fixup → combine with previous, discard message
drop → delete this commit entirely
exec → run a shell command
## ── Example: squash typo fixes, reword first commit ──
reword a1b2c3d feat: add login form
fixup d4e5f6g fix: typo in label
fixup g7h8i9j fix: another typo
pick j1k2l3m feat: add remember me checkbox
drop m4n5o6p chore: console.log cleanup
## ── After save, result: clean 2-commit history ──
* x1y2z3a feat: add login form (polished)
* b4c5d6e feat: add remember me checkbox
bash — exec, autosquash, and fixup! workflow
## ── exec: run tests after each rebased commit ──
pick a1b2c3d feat: add auth
exec npm test
pick d4e5f6g feat: add profile
exec npm test
## ── Autosquash workflow (professional pattern) ──
# Commit a quick fix with special prefix
$ git commit -m "fixup! feat: add login form"
# Run rebase with --autosquash — Git finds the right commit
$ git rebase -i --autosquash HEAD~4
# Git automatically places the fixup! under the right commit!
## ── Global autosquash by default ──
$ git config --global rebase.autoSquash true
# Now -i always autosquashes when prefix matches

Advanced Merge Strategies CONTROL
ours, theirs, octopus, squash — choose your merge weapon.
bash — Merge Strategy Options
## ── ort (default) — smart 3-way merge ──
$ git merge feature/auth -s ort
## ── ours — IGNORE the other branch completely ──
# Keep our version entirely, record merge happened
$ git merge -s ours old-feature
## ── Conflict resolution: prefer ours/theirs ──
# Accept ALL conflicts using our version
$ git merge feature/redesign -X ours
# Accept ALL conflicts using their version
$ git merge feature/redesign -X theirs
## ── Per-file conflict resolution ──
$ git checkout --ours style.css # keep our style.css
$ git checkout --theirs package.json # take their package.json
$ git add style.css package.json
## ── octopus — merge multiple branches at once ──
$ git merge feature/a feature/b feature/c -s octopus
## ── squash merge — flatten all commits into one ──
$ git merge --squash feature/messy-branch
$ git commit -m "feat: clean squashed feature"
## ── rerere — reuse recorded resolution ──
# Enable: Git remembers how you resolved conflicts
$ git config --global rerere.enabled true
# Next time same conflict appears, Git resolves it automatically!
Strategy/OptionWhen to UseResult
--ff-onlyOnly merge if fast-forward possibleNo merge commit, clean line
--no-ffAlways create merge commitExplicit branch boundary
--squashMessy feature branch, clean history wantedOne commit, all changes
-X oursOur version always wins conflictsFast conflict resolution
-X theirsIncoming branch always wins conflictsFast conflict resolution
rerereLong-lived feature branch, repeated rebasesAuto-resolved conflicts

Git Submodules COMPLEX
Nest one repository inside another — with its own independent history.
📦 Submodule Use Cases

Use submodules when you need to include an external repo at a specific commit — shared UI component library, a design system, a vendored dependency with custom patches, or a private SDK. The parent repo records exactly which commit of the submodule to use.

bash — git submodule
## ── Add a submodule ──
$ git submodule add https://github.com/team/ui-lib.git src/lib/ui
Cloning into '/project/src/lib/ui'...
# Two new things created:
.gitmodules ← tracks submodule config
src/lib/ui ← the submodule directory (a pointer, not files)
## ── Clone a repo WITH its submodules ──
$ git clone --recurse-submodules https://github.com/you/project.git
# OR if already cloned without submodules:
$ git submodule update --init --recursive
## ── Update all submodules to latest remote commit ──
$ git submodule update --remote --merge
## ── Run a command in all submodules ──
$ git submodule foreach 'git pull origin main'
$ git submodule foreach 'npm install'
## ── See submodule status ──
$ git submodule status
+a1b2c3d src/lib/ui (v2.1.0-4-ga1b2c3d)
# + means ahead of recorded commit
## ── Remove a submodule (multi-step!) ──
$ git submodule deinit -f src/lib/ui
$ git rm -f src/lib/ui
$ rm -rf .git/modules/src/lib/ui
🚨 Submodule gotcha: After pulling changes that update a submodule pointer, run git submodule update --init --recursive or your submodule will be stuck at the old commit. Many devs forget this and get confused.

Git Subtree SIMPLER ALT
Embed a repo inside another — without the complexity of submodules.
Submodules
  • Separate history maintained
  • Clear boundary between repos
  • Requires --recurse-submodules
  • Extra submodule update after pull
  • Confusing for new contributors
  • Must manage .gitmodules carefully
Subtree
  • Regular git clone just works
  • No extra steps for teammates
  • Can contribute back upstream
  • Merged history can be noisy
  • Slightly complex push-back syntax
  • Harder to keep in sync
bash — git subtree
# Add a remote repo as a subtree under /src/utils
$ git remote add utils-remote https://github.com/team/utils.git
$ git subtree add --prefix=src/utils utils-remote main --squash
Added dir 'src/utils'
# Pull latest changes from the subtree remote
$ git subtree pull --prefix=src/utils utils-remote main --squash
# Push changes you made locally BACK to the upstream repo
$ git subtree push --prefix=src/utils utils-remote feature/my-contribution
# Split a subdirectory OUT into its own repo history
$ git subtree split --prefix=src/utils -b split-utils
$ cd .. && git clone project new-utils-repo
$ cd new-utils-repo && git checkout split-utils

Git LFS — Large File Storage MEDIA
Store large binaries (images, videos, models) outside the Git object store.
bash — git-lfs setup and usage
# Install (once per machine)
$ git lfs install
Git LFS initialized.
# Track large file types (adds to .gitattributes)
$ git lfs track "*.psd"
$ git lfs track "*.mp4"
$ git lfs track "*.ai"
$ git lfs track "public/assets/**"
$ cat .gitattributes
*.psd filter=lfs diff=lfs merge=lfs -text
*.mp4 filter=lfs diff=lfs merge=lfs -text
# Commit .gitattributes first!
$ git add .gitattributes && git commit -m "chore: add LFS tracking"
# Now add large files normally — LFS handles them
$ git add design/mockup.psd && git commit -m "chore: add design mockup"
# Git stored a tiny pointer, LFS stored the actual file!
# See LFS status
$ git lfs status
Objects to be committed: design/mockup.psd (LFS: 24.5 MB)
# See all tracked LFS files
$ git lfs ls-files
a1b2c3d * design/mockup.psd
# Migrate existing large files to LFS retroactively
$ git lfs migrate import --include="*.psd,*.mp4" --everything

Sparse Checkout & Partial Clone MONOREPO
Check out only the files you need — essential for giant monorepos.
bash — Sparse Checkout + Partial Clone
## ── Partial Clone (only download objects you need) ──
$ git clone --filter=blob:none https://github.com/org/monorepo.git
# Downloads commits + trees, blobs fetched on demand
$ git clone --filter=tree:0 https://github.com/org/monorepo.git
# Even lighter: treeless clone
## ── Sparse Checkout (only check out specific dirs) ──
$ git clone --no-checkout https://github.com/org/monorepo.git
$ cd monorepo
$ git sparse-checkout init --cone
$ git sparse-checkout set packages/frontend packages/shared
Updating files: 100% (842/842), done.
# Now you only have frontend/ and shared/ checked out
# packages/backend doesn't exist locally at all!
# Add another package later
$ git sparse-checkout add packages/api
# See what's checked out
$ git sparse-checkout list
packages/frontend
packages/shared
packages/api
# Disable sparse checkout (get everything)
$ git sparse-checkout disable

Prove every commit is really from you — cryptographically.
bash — GPG Key Setup & Commit Signing
# Generate a GPG key
$ gpg --full-generate-key
# Choose: RSA, 4096 bits, your name + GitHub email, no expiry

sec rsa4096/3AA5C34371567BD2 2024-01-01

CI/CD Pipeline with GitHub Actions PRO
Git events trigger automated test, build, and deploy pipelines.
.github/workflows/ci.yml — Full CI/CD Pipeline
name: CI/CD Pipeline
on:
push:
branches: [main, develop]
pull_request:
branches: [main]
release:
types: [published]
jobs:
test:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
with: {node-version: '20'}
- run: npm ci
- run: npm test -- --coverage
- run: npm run lint
deploy-staging:
needs: test
if: github.ref == 'refs/heads/develop'
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- run: npm ci && npm run build
- run: npx vercel deploy --token ${{ secrets.VERCEL_TOKEN }}
deploy-production:
needs: test
if: github.event_name == 'release'
- run: npx vercel --prod --token ${{ secrets.VERCEL_TOKEN }}
🔑 Git event triggers: push to a branch, pull_request opened/updated, release published (from a tag), schedule (cron), workflow_dispatch (manual). Each trigger a different automation.

Git Performance & Maintenance SPEED
Keep large repos fast — gc, maintenance, and pack strategies.
bash — Performance & Maintenance Commands
## ── git gc — garbage collect unreachable objects ──
$ git gc
Counting objects: 1247, done.
Compressing objects: 100% (342/342), done.
$ git gc --aggressive # slower, better compression
## ── git maintenance — scheduled background optimization ──
$ git maintenance start
Scheduled git maintenance tasks. (runs in background)
$ git maintenance run --task=gc --task=commit-graph
## ── commit-graph — speed up log/status in huge repos ──
$ git commit-graph write --reachable
# Dramatically speeds up: git log, git status, git push
## ── Clean up old remote-tracking branches ──
$ git remote prune origin
$ git fetch --prune
## ── Shallow clone for CI (much faster) ──
$ git clone --depth 1 https://github.com/org/repo.git
# 10x faster for CI — only needs latest snapshot
# Deepen if needed later:
$ git fetch --unshallow
## ── Find and remove accidentally committed large files ──

Project 1: Full OSS Contribution Workflow PRO
The complete Fork → PR → Sync cycle used by every open-source contributor.
Full Project 🏗️ Contribute to an Open Source Project

Use a real npm package or GitHub repo. Simulate the full OSS workflow: fork, clone, branch, fix/feature, sync with upstream, open a PR with a clean commit history.

1
Fork a repo on GitHub you want to contribute to (e.g. an awesome-list or small utility).
2
Clone your fork: git clone git@github.com:YOUR_USER/repo.git && cd repo
3
Add upstream: git remote add upstream https://github.com/ORIGINAL/repo.git — verify with git remote -v
4
Sync first: git fetch upstream && git rebase upstream/main — always start fresh
5
Branch: git switch -c fix/correct-typo-in-readme
6
Make your changes in small, atomic commits. Use conventional commits format.
7
Clean up history: git rebase -i upstream/main — squash fixup commits, reword if needed
8
Sync again before push: git fetch upstream && git rebase upstream/main — resolve any conflicts
9
Push to your fork: git push origin fix/correct-typo-in-readme
10
Open Pull Request on GitHub. Fill in description, reference any issues (Closes #42), add screenshots if UI change.
11
Requested changes? Add new commits, git push again — PR updates automatically.
12
PR merged! Delete your branch. git switch main && git pull upstream main && git branch -d fix/correct-typo-in-readme
bash — Upstream sync one-liner alias
# Add this alias: sync your fork's main with upstream in one command
$ git config --global alias.sync-fork '!git fetch upstream && git switch main && git rebase upstream/main && git push origin main'
$ git sync-fork
✔ Fetched upstream, rebased, pushed. Fork is up to date.
