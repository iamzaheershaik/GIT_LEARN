Setup & Configuration
Before anything else — tell Git who you are.
💡 What is Git?

Git is a distributed version control system. It tracks changes in your code over time, lets you branch off to experiment, and merge changes back — all without ever losing history. Every developer machine has the full repository, not just a snapshot.

bash — First-Time Git Setup
# Set your name and email (shown in every commit)
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"
# Set default branch name to 'main'
$ git config --global init.defaultBranch main
# Choose your default editor (VS Code)
$ git config --global core.editor "code --wait"
# Verify all global config
$ git config --list
user.name=Your Name
user.email=you@example.com
init.defaultBranch=main
# View config file directly
$ cat ~/.gitconfig
git config --global
Sets config for ALL repositories on your machine. Stored in ~/.gitconfig
git config --local
Sets config only for the current repo. Stored in .git/config
git config --list
Lists ALL active configuration values from all levels.
💡 On Windows, use Git Bash or WSL. On Mac/Linux, the terminal is all you need. Install Git at git-scm.com

# Create a new folder and init a git repo inside it
$ mkdir my-project && cd my-project
$ git init
Initialized empty Git repository in /home/user/my-project/.git/
# See the hidden .git folder (the brain of git)
$ ls -la
drwxr-xr-x 8 user staff 256 .git
# Look inside .git
$ ls .git
HEAD config description hooks info objects refs
bash — Clone an existing repository
# Clone from GitHub (creates a folder with repo name)
$ git clone https://github.com/user/repo.git
Cloning into 'repo'...
remote: Counting objects: 100
Receiving objects: 100% (100/100), done.
# Clone into a custom folder name
$ git clone https://github.com/user/repo.git my-app
# Clone only the latest snapshot (shallow clone, faster)
$ git clone --depth 1 https://github.com/user/repo.git
# Clone a specific branch only
$ git clone -b develop https://github.com/user/repo.git
🔑 The .git Folder

The .git directory is everything. It stores your commits, branches, tags, config — the entire history. Delete it and you lose all Git tracking. Never delete it.

Stage & Commit
The heartbeat of Git — the 3-stage workflow.
Working Directory
your files
──── git add ────►
Staging Area
index / cache
── git commit ──►
Local Repository
.git/objects
── git push ──►
Remote (GitHub)
origin
bash — Stage and Commit Workflow
# Create a file
$ echo "# My App" > README.md
# Stage ONE file
$ git add README.md
# Stage ALL changed files
$ git add .
# Stage with pattern matching
$ git add src/*.js
# Stage interactively (choose hunks)
$ git add -p
# Commit with a message
$ git commit -m "feat: add README file"
[main (root-commit) a1b2c3d] feat: add README file
1 file changed, 1 insertion(+)
# Skip staging — add and commit tracked files at once
$ git commit -am "fix: update readme"
# Fix your last commit message
$ git commit --amend -m "feat: better message"
# Unstage a file (keep changes in working dir)
$ git restore --staged README.md
⚠️ Conventional Commits: Use prefixes like feat:, fix:, docs:, chore:, refactor: in commit messages. This is industry standard and makes history readable.
🔑 .gitignore

Files you never want Git to track. Create a .gitignore in your project root.

.gitignore — Common Patterns
node_modules/ # entire folder
.env # secret keys — NEVER commit this
dist/ # build output
*.log # all .log files
.DS_Store # Mac OS metadata
/coverage # test coverage (leading slash = root only)
# Remove a file that was accidentally committed
$ git rm --cached .env
rm '.env'

05
Branching
The killer feature of Git. Build in isolation.
🌿 What is a Branch?

A branch is a lightweight movable pointer to a commit. When you branch off, you create a new timeline where you can experiment safely. The main branch is just the production timeline. You create feature branches, do work, and merge back when ready.

bash — Branch Commands
# List all local branches
$ git branch
* main
develop
# List remote branches too
$ git branch -a
# Create a new branch
$ git branch feature/user-auth
# Switch to that branch (new way)
$ git switch feature/user-auth
Switched to branch 'feature/user-auth'
# Create AND switch in one command ← Use this always
$ git switch -c feature/payments
Switched to a new branch 'feature/payments'
# Old way (still widely used)
$ git checkout -b feature/payments
# Rename current branch
$ git branch -m feature/payments feature/stripe-payments
# Delete a merged branch (safe)
$ git branch -d feature/old-feature
# Force delete (even if unmerged)
$ git branch -D feature/bad-experiment
💡 Branch Naming Convention: Use feature/login, fix/navbar-bug, hotfix/payment-crash, release/v1.2. Prefix + short description.

Merging
Bring your branches back together.
Fast-Forward
3-Way Merge
Conflicts
Fast-Forward Merge (no divergence)
# main hasn't moved since feature branch was created
# Git just moves the main pointer forward
$ git switch main
$ git merge feature/add-tasks
Updating a1b2c3d..d4e5f6g
Fast-forward
app.js | 15 +++++++++++++++
1 file changed, 15 insertions(+)
# Force a merge commit even on fast-forward
$ git merge --no-ff feature/add-tasks -m "merge: add tasks feature"
3-Way Merge (both branches diverged)
# main has new commits AND feature branch has new commits
# Git creates a new "merge commit" to combine both
$ git switch main
$ git merge feature/delete-tasks
Merge made by the 'recursive' strategy.
app.js | 8 ++++++++
# After merge, delete the feature branch
$ git branch -d feature/delete-tasks
Deleted branch feature/delete-tasks (was g7h8i9j).
Handling Merge Conflicts
$ git merge feature/redesign
CONFLICT (content): Merge conflict in style.css
Automatic merge failed; fix conflicts and then commit.
# Open style.css — you'll see conflict markers:
<<<<<<< HEAD
background: #ffffff;
=======
background: #1a1a2e;
>>>>>>> feature/redesign
# EDIT the file manually — remove markers, keep what you want
# Then stage and commit:
$ git add style.css
$ git commit -m "merge: resolve style conflict"
# Abort a merge if you get scared
$ git merge --abort
07
Remote Repositories
Connecting your local repo to the world.
bash — Managing Remotes
# Add a remote (GitHub repo URL)
$ git remote add origin https://github.com/you/todo-app.git
# List remotes
$ git remote -v
origin https://github.com/you/todo-app.git (fetch)
origin https://github.com/you/todo-app.git (push)
# Rename a remote
$ git remote rename origin upstream
# Change remote URL (switched from HTTPS to SSH)
$ git remote set-url origin git@github.com:you/todo-app.git
# Remove a remote
$ git remote remove origin
# Add upstream for open-source contributions
$ git remote add upstream https://github.com/original-owner/repo.git
🔐 SSH vs HTTPS: Use SSH keys for authentication (git@github.com:user/repo.git). Once set up, no passwords ever. Use ssh-keygen -t ed25519 then add the public key to GitHub Settings → SSH Keys.
08
Push & Pull
Syncing with your remote.
bash — Push & Pull
# Push main to origin for the first time
$ git push -u origin main
Branch 'main' set up to track remote branch 'main' from 'origin'.
# After -u is set, just use:
$ git push
# Push a feature branch
$ git push origin feature/add-tasks
# Pull = fetch + merge (get remote changes)
$ git pull
Updating a1b2c3d..d4e5f6g
index.html | 3 +++
# Pull with rebase (cleaner history)
$ git pull --rebase
# Force push (DANGER: rewrites remote history)
$ git push --force-with-lease
# Delete remote branch
$ git push origin --delete feature/old-feature
🚨 Never force push to main/master on a shared repo. --force-with-lease is safer than --force — it checks if someone else pushed first.
09
Fetch & Sync
Look before you merge.
bash — git fetch
# Fetch downloads changes but does NOT merge
$ git fetch origin
From https://github.com/you/todo-app
a1b2c3d..d4e5f6g main -> origin/main
# Now inspect what arrived
$ git log main..origin/main --oneline
d4e5f6g feat: new dashboard added by teammate
# See the diff
$ git diff main origin/main
# Merge when you're ready
$ git merge origin/main
# Fetch ALL remotes and prune deleted branches
$ git fetch --all --prune
🔑 Fetch vs Pull

git fetch = safe, just downloads. git pull = downloads AND merges immediately. In teams, fetch first, inspect, then merge. Avoid surprises.

Revert & Reset
How to undo — safely and dangerously.
⚠️ The Golden Rule: git revert is always safe. git reset --hard can destroy your work. Know what you're doing.
bash — Undo Operations
## ── git restore: discard working dir changes ──
$ git restore index.html
# Throws away changes in index.html — back to last commit
## ── git restore --staged: unstage ──
$ git restore --staged index.html
## ── git revert: create a NEW commit that undoes a commit ──
# Safe for shared/public branches!
$ git revert a1b2c3d
[main f1e2d3c] Revert "feat: add broken feature"
## ── git reset: move HEAD backward ──
# --soft: keep changes staged
$ git reset --soft HEAD~1
# --mixed (default): keep changes unstaged
$ git reset HEAD~1
# --hard: DESTROY changes (irreversible!)
$ git reset --hard HEAD~1
HEAD is now at d4e5f6g feat: add task list
# Reset to match remote exactly
$ git reset --hard origin/main
CommandStaging AreaWorking DirUse When
reset --softkept stagedkeptWant to re-commit
reset --mixedunstagedkeptWant to re-stage selectively
reset --hardclearedDELETEDNuke everything — careful!
revertnew commitkeptPublic branches — always safe

Stash
Save your work temporarily — without committing.
bash — git stash
# SCENARIO: Working on feature, urgent bug report comes in!
# Save your half-done work
$ git stash
Saved working directory and index state WIP on main: a1b2c3d
# Stash with a descriptive name
$ git stash save "WIP: user auth form"
# Also stash untracked files
$ git stash -u
# Fix the bug, commit, then come back to your stash
$ git stash list
stash@{0}: WIP on main: a1b2c3d — user auth form
stash@{1}: WIP on main: d4e5f6g — dashboard charts
# Pop most recent stash (apply + remove from stack)
$ git stash pop
# Apply a specific stash (keep it in the list)
$ git stash apply stash@{1}
# Delete a specific stash
$ git stash drop stash@{0}
# Delete ALL stashes
$ git stash clear
12
Rebase
Rewrite history for a cleaner timeline.
🎯 Rebase vs Merge

Merge creates a merge commit showing branches joined. Rebase replays your commits on top of another branch — as if you started your feature from the latest main. Result: clean, linear history.

bash — git rebase
# On feature branch — bring in latest main changes
$ git switch feature/add-tasks
$ git rebase main
Successfully rebased and updated refs/heads/feature/add-tasks.
## ── Interactive Rebase: edit last 3 commits ──
$ git rebase -i HEAD~3
# In the editor you see:
pick a1b2c3d feat: add login form
pick d4e5f6g fix: typo
pick g7h8i9j fix: another typo
# Change to squash to combine commits:
pick a1b2c3d feat: add login form
squash d4e5f6g fix: typo
squash g7h8i9j fix: another typo
# Abort if things go wrong
$ git rebase --abort
# Continue after resolving rebase conflicts
$ git rebase --continue
🚨 Never rebase shared/public branches. Rebase rewrites commit hashes. If teammates have your branch, rebasing causes chaos. Only rebase your own local feature branches.

Cherry-Pick
Grab a specific commit from any branch.
bash — git cherry-pick
# SCENARIO: A hotfix was committed to develop, need it in main NOW
$ git switch main
# Pick only the commit with hash a1b2c3d from develop
$ git cherry-pick a1b2c3d
[main g1h2i3j] hotfix: fix payment null check
1 file changed, 3 insertions(+), 1 deletion(-)
# Cherry-pick multiple commits
$ git cherry-pick a1b2c3d d4e5f6g
# Cherry-pick a range of commits
$ git cherry-pick a1b2c3d^..g7h8i9j
# Cherry-pick without committing (apply only)
$ git cherry-pick -n a1b2c3d

Bisect & Reflog
Forensics tools — find bugs and recover from disasters.
git bisect
git reflog
git bisect — Binary search for the bug-introducing commit
# SCENARIO: A bug exists now but wasn't there 2 weeks ago
$ git bisect start
$ git bisect bad # current commit is bad
$ git bisect good v1.0.0 # this version was fine
Bisecting: 15 revisions left to test after this
# Git checks out a commit in the middle. Test your app.
# If this one is OK:
$ git bisect good
# If this one has the bug:
$ git bisect bad
# Repeat until Git finds the exact commit:
a1b2c3d is the first bad commit
Author: Dev <dev@company.com>
feat: refactor payment service ← HERE'S YOUR BUG
# Exit bisect mode
$ git bisect reset
git reflog — Your safety net. Nothing is truly lost.
# reflog tracks EVERY move of HEAD
$ git reflog
g7h8i9j HEAD@{0}: commit: feat: user profile page
a1b2c3d HEAD@{1}: reset: moving to HEAD~3
d4e5f6g HEAD@{2}: commit: feat: dashboard v2
h1i2j3k HEAD@{3}: checkout: moving from develop to main
# SCENARIO: Accidentally did git reset --hard, lost commits!
# Find the commit you lost in reflog
$ git reflog
d4e5f6g HEAD@{2}: commit: feat: dashboard v2 ← THERE IT IS
# Recover it!
$ git reset --hard d4e5f6g
HEAD is now at d4e5f6g feat: dashboard v2
# Or recover a deleted branch via reflog
$ git switch -c feature/recovered HEAD@{3}
💡 Reflog is your guardian angel. Git doesn't delete objects immediately — they stay for 90 days. If you broke something, git reflog will almost always save you.

Aliases & Pro Tips
Work 10× faster with shortcuts.
~/.gitconfig — Power User Aliases
# Add these aliases
$ git config --global alias.st status
$ git config --global alias.co checkout
$ git config --global alias.br branch
$ git config --global alias.unstage 'restore --staged'
$ git config --global alias.last 'log -1 HEAD'
$ git config --global alias.lg "log --oneline --graph --all --decorate"
$ git config --global alias.undo 'reset --soft HEAD~1'
$ git config --global alias.nuke 'reset --hard HEAD'
# Usage
$ git st # git status
$ git lg # beautiful graph log
$ git undo # undo last commit, keep changes
# See which branches are already merged
$ git branch --merged
# Clean up all merged branches at once
$ git branch --merged | grep -v main | xargs git branch -d
# Show files changed between two commits
$ git diff --name-only HEAD~5 HEAD

Project 2: Simulated Team Workflow
GitHub Flow — PRs, conflicts, and collaboration.
🚀 GitHub Flow (Industry Standard)

The workflow every modern team uses: main is always deployable. All work happens in feature branches. Changes only enter main via Pull Requests (PRs) — reviewed by teammates.

Full Project 🚀 Simulate a 2-Dev Team with One Repo

Simulate Dev A and Dev B working on the same project by using two different local clones. Learn real-world conflict resolution and PR workflow.

1
Setup: Create a GitHub repo. Clone it twice into different folders — clone-devA/ and clone-devB/
2
Dev A: In clone-devA: git switch -c feature/navbar, build navbar, push branch to GitHub.
3
Dev B: In clone-devB: git switch -c feature/footer, build footer, push branch to GitHub.
4
PR 1: On GitHub — Dev A opens a Pull Request from feature/navbar → main. Merge it.
5
Sync Dev B: git fetch origin && git rebase origin/main — Dev B gets latest main before their PR.
6
Force conflict: Both Dev A and Dev B edit the same line in style.css from their branches. Open two PRs.
7
Resolve: The second PR to merge will show a conflict. Resolve it locally, push again, PR passes.
8
Code Review: Use GitHub's review UI — add comments, request changes, approve. This is real work.
9
Protect main: In GitHub Settings → Branches → add rule: require PR + 1 reviewer before merge.
bash — Full GitHub Flow Command Reference
# The complete GitHub Flow loop for every feature:
① $ git switch main && git pull # start fresh
② $ git switch -c feature/my-feature # branch off
③ ... do work, make atomic commits ...
④ $ git push -u origin feature/my-feature # push branch
⑤ ... open Pull Request on GitHub ...
⑥ ... team reviews and approves ...
⑦ ... PR gets merged via GitHub UI ...
⑧ $ git switch main && git pull # sync locally
⑨ $ git branch -d feature/my-feature # clean up
⑩ Repeat for every feature. 🔄
