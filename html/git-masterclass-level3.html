Git Transport Protocol DEEP
How git push and pull actually move data across the network.
🔌 How Git Talks to Remotes

When you run git push, Git doesn't send files — it sends packfiles containing only the objects the remote doesn't have. The protocol negotiates what each side has, computes a delta, and transfers the minimum data. There are three protocols: Smart HTTP/S (most common), SSH, and Git protocol (git://). Dumb HTTP is deprecated.

Git Notes METADATA
Attach metadata to commits without changing their SHA-1.
📝 Notes Without Amending

Git Notes let you attach arbitrary text to any commit — without changing the commit's hash. Perfect for CI build numbers, code-review approvals, deployment records, or external ticket IDs. Notes live in refs/notes/commits and are separate from commit metadata.

bash — git notes
# Add a note to the latest commit
$ git notes add -m "Reviewed by: Alice, Bob | JIRA: PROJ-142"
# Add a note to a specific commit
$ git notes add -m "CI build #482 passed. Deploy: prod-us-east" a1b2c3d
# Show notes in git log
$ git log --show-notes
commit a1b2c3d
feat: add payment gateway
Notes:
CI build #482 passed. Deploy: prod-us-east
# Edit a note
$ git notes edit a1b2c3d
# Remove a note
$ git notes remove a1b2c3d
# Push notes to remote (not pushed by default!)
$ git push origin refs/notes/commits
# Fetch notes from remote
$ git fetch origin refs/notes/commits:refs/notes/commits
# Use named note namespaces (e.g. for CI vs reviews)
$ git notes --ref ci-builds add -m "build:482:passed" HEAD
$ git notes --ref code-review add -m "approved:alice" HEAD
$ git log --notes=ci-builds --notes=code-review

Git Replace API RARE
Substitute one object for another — transparent history surgery.
bash — git replace — Grafting History
# SCENARIO: Monolith was split. New repo has history from day 1
# but only has 1 root commit. You want to attach old history.
# Step 1: Find the root commit of new repo
$ git log --oneline | tail -1
a1b2c3d chore: initial commit from monolith extract
# Step 2: Find the matching commit in old repo history
$ cd old-monolith && git log --oneline | grep "extract"
d4e5f6g chore: extract payments service
# Step 3: Replace — make old commit appear as parent of new root
$ git replace a1b2c3d d4e5f6g
# Now 'git log' shows full history, transparently!
# List all replacements
$ git replace -l
a1b2c3d → d4e5f6g
# Delete a replacement
$ git replace -d a1b2c3d
# Push replacements so teammates see the grafted history
$ git push origin refs/replace/*
# Permanently bake replacements into real history
$ git filter-repo --replace-refs update-no-add

Patch Workflow EMAIL-BASED
The Linux kernel way — share code without GitHub via email patches.
🩹 Why Patches?

The Linux kernel, Git itself, and many OSS projects use a patch-based workflow. You format commits as .patch files, send them via email (or attachment), and maintainers apply them. This workflow predates GitHub and is still used for projects with thousands of contributors. Also useful for: sending changes over Slack, USB, or air-gapped systems.

bash — git format-patch + git am
## ── Creating Patches ──
# Create a patch file for the last commit
$ git format-patch -1 HEAD
0001-feat-add-payment-gateway.patch
# Create patches for last 3 commits
$ git format-patch -3
0001-feat-login-page.patch
0002-feat-user-profile.patch
0003-feat-payment-gateway.patch
# Patches since branching from main
$ git format-patch main..HEAD
# Patch with cover letter (like a PR description)
$ git format-patch --cover-letter -3 -o patches/
patches/0000-cover-letter.patch ← edit this with description
patches/0001-feat-login.patch
## ── Applying Patches ──
# Apply patch preserving author, date, message
$ git am 0001-feat-add-payment-gateway.patch
Applying: feat: add payment gateway
# Apply all patches in a directory
$ git am patches/*.patch
# Apply fails? Resolve conflict then continue
$ git am --show-current-patch # see what failed
$ git am --continue # after resolving
$ git am --abort # give up
## ── Quick patch (no commit, just the diff) ──
$ git diff HEAD~2 > changes.patch
$ git apply changes.patch # apply without commit
$ git apply --check changes.patch # dry run

Smudge & Clean Filters RARE POWER
Transform file content on checkout and commit — transparently.
🧹 Filters Explained

Clean filter runs when you stage a file — transforms what goes into the repo. Smudge filter runs when you checkout — transforms what appears in your working directory. Together they let you: auto-encrypt secrets, strip notebook outputs, inject version numbers, or normalize whitespace — all transparently.

bash — Smudge/Clean Filter: Auto-encrypt .env secrets
## ── Example: Jupyter Notebook output stripper ──
# Problem: notebook outputs bloat git diff. Strip on commit.
# 1. Define the filter in .gitconfig
$ git config --global filter.nbstrip.clean "jupyter nbconvert --to notebook --ClearOutputPreprocessor.enabled=True - --stdout 2>/dev/null"
$ git config --global filter.nbstrip.smudge cat
$ git config --global filter.nbstrip.required true
# 2. Apply in .gitattributes
*.ipynb filter=nbstrip
## ── Example: Auto-expand version keyword ──
# Store "$VERSION$" in repo, expand to real version on checkout
$ git config filter.version.smudge 'sed "s/\$VERSION\$/$(git describe --tags)/"'
$ git config filter.version.clean 'sed "s/[0-9]\+\.[0-9]\+\.[0-9]\+/\$VERSION\$/"'
## ── Example: Crypt filter (encrypt on commit, decrypt on checkout) ──
$ git config filter.crypt.clean "openssl enc -aes-256-cbc -pbkdf2 -pass env:GIT_CRYPT_KEY"
$ git config filter.crypt.smudge "openssl dec -aes-256-cbc -pbkdf2 -pass env:GIT_CRYPT_KEY"
secrets.yaml filter=crypt # in .gitattributes
# secrets.yaml is encrypted in repo, decrypted in your editor
## ── git-crypt: production-grade encryption ──
$ git-crypt init
$ git-crypt add-gpg-user YOUR_GPG_KEY_ID
secrets/** filter=git-crypt diff=git-crypt

Workflow Wars: Git Flow vs Trunk-Based STRATEGY
When to use which — the definitive comparison with real-world guidance.
Git Flow
Branches: main, develop, feature/*, release/*, hotfix/*

  • Clear separation of stable vs dev
  • Good for scheduled releases (SaaS v2.0)
  • Hotfixes don't disrupt ongoing features
  • Complex — many parallel branches
  • Long-lived branches = merge conflicts
  • Slow to ship — batch releases only

Use when: versioned software, libraries, mobile apps with app-store releases.
Trunk-Based Development
Branches: main only (feature flags for incomplete work)

  • Continuous deployment — ship every commit
  • No merge hell — tiny PRs, fast reviews
  • What Google, Facebook, Netflix use
  • Requires feature flags for in-progress work
  • Needs excellent test coverage
  • Harder for juniors to understand

Use when: web apps, SaaS, startups, CI/CD-mature teams.
bash — git flow CLI tool
# Install git-flow extension
$ brew install git-flow-avh
$ git flow init
Which branch is your production branch? [main]
Which branch is your development branch? [develop]
Feature branches prefix? [feature/]
## ── Feature workflow ──
$ git flow feature start user-auth
# ... do work ...
$ git flow feature finish user-auth
# Merges to develop, deletes feature/user-auth
## ── Release workflow ──
$ git flow release start 2.0.0
# bump version, update changelog, final testing
$ git flow release finish 2.0.0
# Merges to main AND develop, tags v2.0.0
## ── Hotfix workflow ──
$ git flow hotfix start payment-crash
# fix bug on production code
$ git flow hotfix finish payment-crash
# Merges to main AND develop, tags v1.2.1

Custom Git Commands EXTEND GIT
Write git anything — add your own commands to git's vocabulary.
🪄 How Custom Commands Work

Any executable file named git-something on your $PATH becomes git something. Git discovers it automatically. Write in any language — bash, Python, Node. This is how tools like git-flow, git-lfs, and git-extras are built.

bash — Build 3 custom git commands
## ── git standup: show your commits from last 24h ──
$ cat > ~/bin/git-standup
#!/bin/bash
AUTHOR=$(git config user.name)
echo "📋 Your commits since yesterday, $AUTHOR:"
git log --since="yesterday 00:00" --author="$AUTHOR" \
--oneline --no-merges --all
$ chmod +x ~/bin/git-standup
$ git standup
📋 Your commits since yesterday, RyzeN:
a1b2c3d feat: add JWT middleware
d4e5f6g fix: correct token expiry
## ── git nuke-merged: clean all merged branches ──
$ cat > ~/bin/git-nuke-merged
#!/bin/bash
MAIN="${1:-main}"
echo "🧹 Deleting merged branches (except $MAIN)..."
git branch --merged "$MAIN" | grep -v "^\*\|$MAIN\|develop" | xargs -r git branch -d
git remote prune origin
echo "✅ Done."
$ git nuke-merged
## ── git changelog: auto-generate from conventional commits ──
$ cat > ~/bin/git-changelog
#!/bin/bash
FROM="${1:-$(git describe --tags --abbrev=0)}"
echo "## Changelog ($FROM..HEAD)\n"
echo "### Features"
git log $FROM..HEAD --no-merges --format="- %s (%h)" | grep "^- feat"
echo "\n### Bug Fixes"
git log $FROM..HEAD --no-merges --format="- %s (%h)" | grep "^- fix"
$ git changelog v1.2.0 > CHANGELOG.md

Understand Git completely by reimplementing its core in Node.js.
Expert Project 🔬 mini-git: Core Internals in ~200 lines

The ultimate way to understand Git is to build it. Implement: SHA-1 hashing, blob creation, tree objects, commits, and a working index. By the end, you'll understand every git add and git commit at the byte level.

1
Init: mkdir mini-git && cd mini-git. Create mkdir -p .mgit/objects .mgit/refs/heads. Write HEAD: ref: refs/heads/main
2
hash-object: Read a file, prepend blob {size}\0, SHA-1 hash it, zlib-compress it, write to .mgit/objects/ab/cdef...
3
cat-file: Read an object file, decompress with zlib, strip header, print the content — your first plumbing command
4
update-index: Write a staging area (index file) mapping filenames to blob hashes
5
write-tree: Read the index, create a tree object from it (tree header + mode + filename + SHA entries)
6
commit-tree: Create a commit object pointing to a tree SHA, with author, timestamp, and message
7
update-ref: Write the commit SHA to .mgit/refs/heads/main — "moving the branch pointer"
8
log: Read HEAD → read commit → read parent → repeat. Print commit chain. You've built git log.
9
Verify: Run both real git and your mini-git on the same files. Compare SHA hashes — they should match!
10
Reflection: You now understand why git rebase changes SHAs, how pack files work, and why Git is so reliable.
node.js — mini git hash-object (core snippet)
const { createHash } = require("crypto");
const { deflateSync } = require("zlib");
const { readFileSync, mkdirSync, writeFileSync } = require("fs");
function hashObject(filePath) {
const content = readFileSync(filePath);
const header = Buffer.from(`blob ${content.length}\0`);
const store = Buffer.concat([header, content]);
// SHA-1 of the full object (same as git does internally)
const sha1 = createHash("sha1").update(store).digest("hex");
// Write compressed object to .mgit/objects/ab/cdef...
const dir = `.mgit/objects/${sha1.slice(0,2)}`;
mkdirSync(dir, { recursive: true });
writeFileSync(`${dir}/${sha1.slice(2)}`, deflateSync(store));
return sha1;
}
// Test: should produce SAME SHA as real git!
$ node -e "const h=require('./hash'); console.log(h.hashObject('README.md'))"
8ab686eafeb1f44702738c8b0f24f2567c36da6d
$ git hash-object README.md
8ab686eafeb1f44702738c8b0f24f2567c36da6d ← identical!
